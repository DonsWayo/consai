# Consai — Design

**A macOS menu-bar-first app for Apple's `container`.**
Status: design approved · 2026-06-17

---

## 1. Summary

Consai is a native macOS menu bar app for managing Apple Containers (Apple's
[`container`](https://github.com/apple/container) tooling). Clicking the menu bar
icon opens a compact panel listing your containers, **grouped into compose stacks**,
with quick actions. Heavy tasks (log streaming, creating a container) open
dedicated windows.

It is the lightweight, menu-bar-first counterpart to full windowed GUIs like
[Orchard](https://github.com/andrew-waters/orchard). It is designed so its core
engine (`ConsaiCore`) can later power a full windowed app — the Docker Desktop
pattern: a menu bar app **and** a full app sharing one engine.

### Positioning (why this app exists)

Research (2026-06) found the surrounding space is crowded *except* for the menu
bar niche:

- **Compose for Apple container already exists** — a dozen+ implementations.
  The de-facto standard is
  [Mcrich23/Container-Compose](https://github.com/Mcrich23/Container-Compose)
  (`brew install container-compose`). **We do not build our own compose engine.**
- **Full windowed GUIs already exist** — Orchard, AppleContainerGUI, ContainerUI,
  apple-container-desktop.
- **A polished menu-bar-first app for Apple `container` does not exist.** The
  Docker-world menu bar apps (Captain, OrbStack) target Docker, not Apple
  `container`. Orchard only has menu bar *integration* as a side feature.

**That gap is the opportunity.** We integrate an existing compose engine and focus
entirely on a great menu-bar-first experience.

## 2. Scope

### v1 (in scope)

- Menu bar panel: live list of containers with status dots, **grouped by stack**
  (compose project) plus standalone containers.
- Quick actions per container: start / stop / restart / delete.
- Per-stack actions: `up` / `down` (via `container-compose` CLI), start/stop all,
  reveal compose file.
- **Log window** (per container) — opens from the panel, tails `container logs -f`.
- **Create-container window** — simple form (image, name, env, ports, volumes).
- Menu bar status: icon/badge reflecting "system service running?" + running count.
- Settings window: check/start/stop the `container` system service; configure the
  `container-compose` binary path; poll interval.

### Deferred (v2+)

- Images tab (list / pull / delete), Networks, Volumes, Builders.
- Multi-pane / multi-container log viewer (Orchard-style).
- The future full windowed app.
- Real-time CPU/mem stats graphs.

The v1 spine — the SDK layer, the stack-grouping model, the windowing pattern, the
service-health check — is what every later feature reuses.

## 3. Architecture

One Xcode project (generated by **XcodeGen** from `project.yml`) containing a local
Swift package for the core:

```
Consai.xcodeproj           (generated by `xcodegen generate`)
├─ ConsaiCore  (Swift package — reusable engine, NO SwiftUI)
│   ├─ Models               Consai value types (Container, Stack, ServiceStatus…)
│   ├─ ContainerEngine      wraps apple/container SDK (list/start/stop/delete/create, stats)
│   ├─ ComposeEngine        wraps `container-compose` CLI (up/down) via Foundation.Process
│   ├─ ProjectRegistry      persists {projectName → composeFilePath}; folds containers into stacks
│   └─ ServiceHealth        is the `container` system service running? start/stop it
└─ Consai (app target — SwiftUI, LSUIElement=true, non-sandboxed)
    ├─ AppState             observable; owns engines; polls for updates
    ├─ MenuBarExtra panel   lists + quick actions
    └─ Windows              LogWindow, CreateContainerWindow, SettingsWindow
```

**Key principles**

- `ConsaiCore` has **no UI** — it is the piece the future full app reuses. The UI
  talks to it only through narrow protocols (`ContainerEngine`, `ComposeEngine`,
  `ServiceHealthChecking`), so the SDK/CLI can be swapped or mocked in tests.
- The app target is a thin SwiftUI layer: `MenuBarExtra` for the panel,
  `Window`/`WindowGroup` for pop-out windows, one `AppState` owning the engines.
- **Non-sandboxed**, distributed outside the App Store (Homebrew/release), exactly
  like Orchard — required to reach the XPC daemon and spawn `Process`.

## 4. Backend & dependencies

- **Raw management → `apple/container` Swift SDK (typed, over XPC).** Proven
  consumable: Orchard depends on the `ContainerAPIClient` SPM product from
  `github.com/apple/container.git`. API surface confirmed:
  `ContainerClient.list(filters:) -> [ContainerSnapshot]`, plus `NetworkClient`,
  `ClientVolume`, images client; models `ContainerSnapshot`, `ContainerStats`,
  `ContainerCreateOptions`, `PublishPort`, `ContainerStatus`.
- **Compose → shell out to `container-compose` CLI** (`up`/`down`) via `Process`.
  Mcrich23/Container-Compose has a `ContainerComposeCore` target but declares **no
  library product**, so it cannot be imported via SPM today. Shelling out keeps zero
  coupling. (Future option: upstream a one-line `.library` product and swap to a
  library call behind the same `ComposeEngine` protocol — no UI change.)
- Build/runtime requirements: **macOS 26 (Tahoe)**, **Xcode 26 / Swift 6.2**,
  `container` installed, `container-compose` installed (optional — only for stacks).

## 5. Data flow & live updates

The SDK is request/response (`list()` is a poll, not a push stream), so:

- **Adaptive polling.** Panel open → poll `ContainerEngine.list()` every **~2s**;
  panel closed → every **~15s** (just to keep the menu bar icon/badge honest).
- **After an action** → immediate re-poll + an **optimistic UI update** so status
  reacts instantly; revert on failure.
- **One source of truth.** `AppState` holds the latest `[Container]` and `[Stack]`;
  the panel, windows, and menu bar icon all render from it. No view fetches on its own.
- **Stack assembly each poll.** Raw `[Container]` → `ProjectRegistry` folds them into
  `[Stack]` by `<project>-<service>` prefix (+ the registry of stacks we launched);
  leftovers become standalone. Pure, testable, no I/O.
- **Logs are a separate stream** — the log window tails `container logs -f <name>` as
  its own `Process`, cancelled when the window closes.

Defaults: poll 2s / 15s (later configurable); optimistic updates ON.

## 6. Compose / stack lifecycle

**How stacks are identified — by name prefix, NOT label.** `container-compose` names
containers **`<project>-<service>`** (project = compose file's `name:` field, else the
directory name). Its own source confirms it does **not** stamp a project label and does
**not** isolate networks/volumes per project. So Consai groups by the
`<project>-<service>` prefix and keeps its own **project registry** for robustness.

**Up:**
1. User picks a compose file (file picker, or a remembered "recent" list). Project name
   follows the same rule `container-compose` uses, so our prefix matches its names.
2. `ComposeEngine.up(file:)` runs `container-compose up -d` with the file's directory as
   `cwd` (detached, so Consai doesn't own the lifetime).
3. On success, `ProjectRegistry` records `{ projectName → composeFilePath, lastUp }`
   (persisted as JSON in Application Support).
4. Next poll folds the new `<project>-<service>` containers under the stack.

**Show:** collapsible group header (`▸ myapp · 3/3 running`) with per-service rows.
Header actions: up / down / restart-all / reveal compose file. Row actions: start /
stop / restart / logs.

**Down:** `ComposeEngine.down(file:)` runs `container-compose down` in the project's
`cwd`. The registry entry is kept (shows as a *known, stopped* stack you can re-`up`)
unless the user removes it.

**Reconciliation (real world).** Grouping is recomputed from live `list()` every poll,
so the panel self-heals when containers appear/vanish outside Consai:
- Match a *known* project prefix → grouped under it.
- An unknown `<x>-<y>` group → shown as an **inferred stack**, visually marked (Consai
  didn't launch it, may lack its compose file, so down/re-up may be unavailable until
  the user points us at the file).
- No matching prefix → **standalone**.

**Honest limitation, surfaced in UI.** Because grouping is name-based and
`container-compose` doesn't isolate per project, a standalone container coincidentally
named `foo-bar` could be mis-grouped. We mark inferred stacks distinctly and **never
destroy a resource on a `down` we can't tie to a known compose file.** No silent
guessing that deletes things.

## 7. Error handling

- **`container` service down** → `ServiceHealth` detects; menu bar dot goes amber;
  panel shows a banner *"Container service not running — [Start]"* → runs
  `container system start`. Container actions disabled until healthy.
- **`container-compose` not installed** → probed at launch; stack up/down actions show
  *"Install container-compose"* with the brew hint. **Raw container management still
  works fully** — compose is an optional capability.
- **SDK/XPC call fails** → engines throw a typed `ConsaiError`; `AppState` shows a
  transient toast and reverts the optimistic update.
- **`compose up`/`down` fails** → capture `stderr` and show it in a sheet, so the user
  sees *why* (bad file, port clash) instead of a dead spinner.
- **No zombie processes** → every spawned `Process` (`logs -f`, compose) is tracked and
  terminated on window-close / app-quit.

## 8. Testing

- **`ConsaiCore` is pure + protocol-based** → high-value logic unit-tested with no I/O:
  `ProjectRegistry` grouping (prefix folding, inferred-vs-known, standalone leftovers)
  against fixture lists; `ComposeEngine` **argument building** (assert `Process` args
  without running); `AppState` poll/diff + optimistic-revert against a **mock**
  `ContainerEngine`.
- **Integration smoke tests** are gated — run only where `container` +
  `container-compose` exist (CI runs just the pure `ConsaiCore` suite).
- **UI** → SwiftUI previews + manual verification for v1; snapshot tests later.
